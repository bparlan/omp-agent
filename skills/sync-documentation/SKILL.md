# Documentation Sync: Distill Engineering Work to Canonical Documents

You are a documentation maintainer that updates project documentation to reflect completed engineering work.

## When to Invoke

Only execute when explicitly requested by user. Not automatic.

## Your Process

1. **Read completed artifacts** — Load Milestone `M{X}.md`, Specifications `M{X}S{Y}.md`, Verification `M{X}S{Y}V.md`, Review Report `M{X}S{Y}R.md`, Completion Report `M{X}S{Y}C.md`, Summary `M{X}S{Y}S.md`, Investigation Report `M{X}S{Y}I{Z}.md`, and `EXPERIENCES.md`.
2. **Examine repository state** — Use `glob`/`grep` to find current documentation and understand structure.
3. **Check for /docs/ingest/ entries** — If session-audit generated INGEST_ENTRIES.md, handle accordingly:
	-   Read INGEST_ENTRIES.md
	-   Present files to user with ask tool
	-   Ask for permission to process
	-   Ask for context (what skill/prompt to use)
	-   Delegate to appropriate skill
	-   Archive original file after processing
	- Load reports from `docs/hotfixes/`.
4. **Identify material changes** — Determine if any canonical documents need updates based on Change Detection criteria.
4a. **Diagram Alignment** — Compare the updated canonical documents with the visual snapshots in `docs/diagrams/`. If structural changes occurred (e.g., new skills added, pipelines altered, templates modified), invoke the `diagrammer` skill to regenerate the affected diagrams ensuring visual state matches codebase reality.
5. **For each documentation file:**
	-   Compare current content against repository reality
	-   Determine if material change occurred
	-   If yes: produce minimal, incremental edits
	-   If no: leave unchanged
6. **Preserve style** — Match existing writing conventions.
7. **Avoid duplication** — Summarize, don't copy full spec/milestone content.
8. **Propose Roadmap Updates** — Based on the completed artifacts, technical debt noted in the Review Report, or obvious logical next steps, propose 1-3 specific updates to `ROADMAP.md`. Ask the user for approval. If approved, use `edit` to add these to the `ROADMAP.md` future items list.
9. **Write revision summary** — Use the summary template.

## File Locations and Workflow

- **Milestone artifacts** (M{X}SA{Y}.md, SESSION_CHANGES.md, CHANGELOG_ENTRIES.md, MILESTONE_UPDATES.md) MUST go to `milestones/M{X}/` or `milestones/TEMP/`
- **INGEST_ENTRIES.md** MUST go to `milestones/M{X}/` or `milestones/TEMP/`
- **`/docs/ingest/`** folder is ONLY for temporary files that will be archived after processing by manage-roadmap
- **Do NOT copy milestone artifacts to /docs/ingest/**
- **Do NOT copy skill files to /docs/ingest/** — process them from their source locations
- When processing files from `INGEST_ENTRIES.md`:
	- Read the file from its original location (e.g., `skills/session-audit/SKILL.md`)
	- Document the changes (update `CHANGELOG.md`, `MILESTONES.md`, etc.)
	- Do NOT copy the file to `/docs/ingest/` before or after processing

## Ingestion Processing Workflow

When INGEST_ENTRIES.md exists (generated by session-audit):

1. **Read INGEST_ENTRIES.md**:
	```bash
	cat milestones/M{X}/INGEST_ENTRIES.md
	```

2. **Display files to user**:
	- List all modified files
	- Show file paths
	- Indicate which files need processing

3. **Ask for permission**:
	- Use `ask` tool to present options:
	```
	Permission required to process milestone files.
	Please confirm which files to process.
	Options:
	- All files (recommended)
	- Specific files only
	- Cancel processing
	```
	- If user cancels, stop processing

4. **Ask for context** (if user approves):
	- Use `ask` tool to get processing context:
	```
	What skill/prompt should process these files?
	Options:
	- evolve-skills (for framework improvements)
	- implement-specification (for new features)
	- generate-spec (for specs)
	- generate-verification (for verification protocols)
	- Other (please specify)
	```

5. **Delegate to appropriate skill**:
	- Read each file from its original location (e.g., `skills/session-audit/SKILL.md`)
	- Document the changes (update CHANGELOG.md, MILESTONES.md, etc.)
	- Do NOT copy files to /docs/ingest/
	- Use IRC to signal completion to manage-roadmap
6. **Archive processed files**:
	- After successful processing, archive original files from milestone folder:
	```bash
	mkdir -p milestones/M{X}/archived
	mv {filename} milestones/M{X}/archived/{filename}
	```
	- Update INGEST_ENTRIES.md to mark as processed
	- Record timestamp and processing skill in INGEST_ENTRIES.md

## Change Detection

Update documentation only when material changes occurred:

| Triggers documentation update | Ignored items          |
| ----------------------------- | ---------------------- |
| Architecture changes          | Refactoring            |
| Public API changes            | Formatting changes     |
| Repository structure changes  | Implementation details |
| Configuration changes         | Temporary experiments  |
| Operational procedures        | Review comments        |
| User-visible behavior         | Internal discussions   |
| Deployment changes            |                        |
| Workflow changes              |                        |
| Supported platforms           |                        |
| Ingestion entries processed   | Cosmetic changes       |
| Architectural Drift             | Triggers documentation update AND diagrammer regeneration |

For `Hotfix Focus` reports, ONLY update the `CHANGELOG.md` or canonical docs if the `Documentation Impact` is MODERATE or MAJOR. If it is MINOR, ignore it entirely.

## Documentation Principles

- **Describe reality** — Documentation reflects actual repository state
- **Summarize, don't copy** — Extract essence, avoid duplication
- **Remove obsolete** — Delete outdated information
- **Be concise** — Every line should add value
- **Match style** — Preserve existing tone and conventions
- **Incremental updates** — Prefer targeted edits over rewrites

## For Each Document

### MILESTONES.md
- Update milestone status from active to archived/completed
- Add links to archived milestone artifacts
- Note milestone dependencies and relationships

### AGENTS.md
	- `Consumer AGENTS.md (~/devcode/aef/AGENTS.md)`: Update ONLY when new Spec-Driven Development (SDD) pipeline rules, consumer-facing skills, artifact routing standards, or project-level coding conventions are discovered.
	- `Dev AGENTS.md (~/devcode/aef/agent/AGENTS.md)`: Update ONLY when new framework infrastructure editing rules (like SWAP.BLK syntax), template versioning requirements, internal directory pathing, or OMP tool usage rules are introduced.

### DATA.md
- Update data models if schema changed
- Add configuration changes
- Note data flow modifications

### ROADMAP.md
- Add completed milestones as done items
- Archive completed features
- Add user-approved proposals for future work, technical debt resolution, or optimizations.

### SPEC.md
- Update only for new/changed architecture
- Add/update public interfaces
- Remove obsolete modules/interfaces

### FRAMEWORK.md
- Update workflows if changed
- Note new patterns or conventions
- Remove deprecated practices

### PLAYBOOK.md
- Add new procedures
- Update operational steps
- Remove obsolete commands

### CHANGELOG.md
- Append completed milestone summaries
- Note version bumps
- Record public API changes
- Document breaking changes
- Add session-audit workflow if major changes occurred

### README.md
- Update usage if user behavior changed
- Note new capabilities
- Remove deprecated features
- Document session-audit process
- Document code-search integration

### EXPERIENCES.md
- Read active friction points and applied skill updates
- If EXPERIENCES.md contains project-specific architectural friction, distill those learnings into `AGENTS.md` or `PLAYBOOK.md` to establish new coding conventions

### SKILLS.md
- Update skill count and layer classifications when new skills are added or existing skills change category
- Add new skills to catalog with description from their SKILL.md frontmatter
- Document skill dependencies and relationships
- Note deprecated or archived skills
